# 持续 Agent 循环

用这个 skill 设计“持续跑”的执行流，而不是一次性交互。目标是让循环可控、可停、可恢复。

## 何时使用

- 任务需要多轮自动推进
- 需要把研究、实现、验证拆成循环
- 需要并行 agent 分发
- 需要明确停止条件、失败恢复和质量门

## 选择路径

```text
开始
 |
 +-- 只是固定顺序步骤? ------> 顺序流水线
 |
 +-- 需要并行探索/生成? -----> 并行循环
 |
 +-- 需要长期自治推进? -----> 持续循环
```

## 最小闭环

每个 loop 至少包含这 5 段：

1. 输入：本轮目标和边界
2. 执行：单 agent 或多 agent
3. 质量门：测试、审查、验证
4. 记录：产出、失败原因、下一轮入口
5. 停止判断：完成 / 阻塞 / 超预算 / 需人工确认

## 必须有的保护栏

- 明确最大轮数
- 明确预算或时间上限
- 明确质量门失败后的动作
- 明确哪些情况必须停下来问人
- 明确不会在同一错误上无限重试

## 常见失败模式

- 一直循环，但没有可衡量进展
- 失败后只是原样重试
- 子任务拆得太碎，协调成本超过收益
- loop 能做很多事，但没有一个可靠出口

## 恢复策略

- 缩小范围，只重跑失败单元
- 提高约束，补充明确验收标准
- 从并行改回顺序，减少变量
- 触发人工审查，而不是继续盲跑

## 何时不要使用

- 单次对话就能完成的小任务
- 需求边界还没清楚的任务
- 没有验证手段的任务

- 先做任务拆分时，配合 `dev-writing-plans`
- 进入实现与验证时，配合 `dev-executing-plans`、`dev-verification`
