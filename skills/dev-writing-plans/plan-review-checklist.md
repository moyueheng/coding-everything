# 实施计划审查清单

在 `dev-writing-plans` 中，每写完一段计划，都必须按这份清单审查后再继续。

## 审查目标

- 计划是否与设计一致
- 任务是否已经拆到可执行粒度
- 文件边界和验证路径是否明确
- 执行者是否可以直接照做

## 必查项

| 类别 | 要检查什么 |
|------|------------|
| 设计一致性 | 是否和设计文档一致，是否有 scope creep |
| 粒度 | 是否已拆到 2-5 分钟级别，是否仍有“大步” |
| 文件边界 | 文件职责是否清晰，是否存在跨模块隐式耦合 |
| 路径精确性 | 创建/修改/测试文件路径是否精确到可执行 |
| TDD 闭环 | 涉及代码的任务是否都有 RED / 验证 RED / GREEN / 验证 GREEN / REFACTOR |
| 验证方式 | 是否给出精确命令和预期输出 |
| 高风险拆分 | 不确定项、高风险项是否被单独拆开 |
| 可执行性 | 执行者是否无需补全上下文即可直接行动 |

## 红旗

- 写“类似前面那一步”但没有具体内容
- 任务同时修改多个大模块
- 没写验证命令或预期结果
- 代码任务没有明确 `dev-tdd` 闭环
- 文件路径模糊，如“更新相关文件”
- 把待决策的问题留给执行阶段现场判断

## 通过标准

只有同时满足下面条件，当前计划段落才算通过：

- 与设计文档一致
- 粒度足够小
- 路径、命令、预期输出明确
- 没有明显隐式耦合
- 执行者可以直接交给 `dev-executing-plans`
