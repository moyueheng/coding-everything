# 设计文档审查清单

在 `dev-brainstorming` 完成设计文档后，必须按这份清单做一次怀疑式审查。

## 审查目标

- 设计是否完整
- 边界是否清楚
- 是否和现有代码库约束冲突
- 是否存在过度设计或错误重造轮子

## 必查项

| 类别 | 要检查什么 |
|------|------------|
| 完整性 | 是否有 TODO、占位符、TBD、"后续再定" |
| 约束覆盖 | 是否遗漏边界条件、失败路径、异常流、集成点 |
| 一致性 | 是否存在前后矛盾、术语漂移、互相冲突的要求 |
| 清晰度 | 是否有执行者必须脑补才能理解的描述 |
| YAGNI | 是否引入用户未要求的功能、过早抽象、额外层级 |
| 复用判断 | 是否把应复用/改造已有方案的问题，错误设计成重新实现 |
| 架构边界 | 单元职责是否单一，接口是否明确，是否存在隐式耦合 |
| 贴合现状 | 是否和当前目录结构、已有模式、项目约定冲突 |

## 红旗

- 某一节明显比其他部分更空
- 写了“实现时再决定”
- 没写错误处理和回滚路径
- 同一个模块承担多个责任
- 修改范围跨多个模块，但没有解释边界
- 没证明为什么不能复用现有实现或现有工具

## 通过标准

只有同时满足下面条件，设计文档才算通过：

- 没有 TODO / TBD / 占位符
- 边界、依赖、失败路径明确
- 没有明显过度设计
- 与当前仓库结构和规则一致
- 执行者无需额外脑补即可进入 `dev-writing-plans`
